# 质量审查 - Alex 的审稿台

## 概述
对完成的论文执行多层质量审查，发现问题后自动修复并重审，循环直到通过或达到上限。这是科研军团的最后一道质量防线，确保交付物达到投稿标准。以 Alex 的视角执行（参考 ~/.claude/agents/alex.md）。

## 常量

```
MAX_ROUNDS = 3                # 最大审查轮数
PASS_SCORE = 70               # 通过分数线（B 级）
BLOCK_OVERRIDE = false        # BLOCK 级问题不可被覆盖
STATE_FILE = "REVIEW_STATE.json"  # 状态持久化文件
```

## 流程

### Step 0: 状态恢复检查
1. 检查是否存在 `REVIEW_STATE.json`
2. 若存在，读取上一轮审查状态，从中断轮次继续
3. 若不存在，从第 1 轮开始

### Step 1: 八层审查矩阵

对 `manuscript.md` 执行以下八个维度的审查：

#### 1.1 数据可追溯性（权重 20%）
- 正文中的每个数字是否都能在 analysis_results.md 中找到对应来源
- 图表中的数据是否与正文描述一致
- 样本量在各章节中是否一致
- 百分比之和是否为 100%（分类变量）

#### 1.2 图表规范（权重 10%）
- 分辨率 ≥ 300 DPI
- 色盲友好配色
- 图例完整且可独立阅读
- 所有图表在正文中按顺序首次引用

#### 1.3 统计标准（权重 20%）
- 效应量和置信区间是否同时报告
- P 值报告格式是否规范（精确值，非 P<0.05）
- 多重比较是否做了校正
- 统计方法与数据类型是否匹配
- 敏感性分析是否充分

#### 1.4 格式规范（权重 5%）
- IMRAD 结构完整
- 缩写首次使用时有全称
- 参考文献格式统一
- 字数在目标期刊范围内

#### 1.5 投稿完整性（权重 10%）
- 标题、摘要、正文、参考文献齐全
- 作者信息、通讯作者、基金声明到位
- STROBE/CONSORT 清单（视研究类型而定）
- 利益冲突声明

#### 1.6 文-图-表同步（权重 15%）
- 正文描述与图表展示是否一致
- 摘要中的数字与正文是否一致
- 结论是否有结果支撑（不过度解读）

#### 1.7 术语一致性（权重 5%）
- 同一概念全文使用同一术语
- 变量名与 data_dictionary.md 一致
- 疾病/药物名称使用标准医学术语

#### 1.8 创新性评估（权重 15%）
- 研究问题是否有新意
- 方法学是否有改进
- 发现是否对临床实践有实际指导意义
- 与同领域最新研究的差异化

### Step 2: 问题分级

每个发现的问题按严重程度分级：

| 级别 | 定义 | 处理 |
|------|------|------|
| BLOCK | 学术诚信问题 / 数据错误 / 统计方法根本错误 | **必须修复**，不可覆盖 |
| WARN | 影响论文质量但不致命 | Wei（协调者）决定是否修复 |
| INFO | 改进建议，可选 | 记录但不强制 |

**特别规则**：学术诚信问题（数据伪造嫌疑、剽窃、不当署名）一律 BLOCK，无例外。

### Step 3: 评分计算

```
总分 = Σ(各维度得分 × 维度权重)
每个维度: 0-100 分

等级:
  A: ≥ 85  优秀，可直接投稿
  B: ≥ 70  良好，微调后可投稿
  C: ≥ 55  及格，需较大修改
  D: < 55  不及格，需重大修改
```

### Step 4: 自动修复循环

```
当前轮次 = 1

while 当前轮次 ≤ MAX_ROUNDS:
    执行八层审查 → 生成问题清单
    计算总分

    if 总分 ≥ PASS_SCORE 且 无 BLOCK 级问题:
        通过 → 进入下一阶段
        break

    对所有 BLOCK 问题执行自动修复:
        - 数据不一致 → 以 analysis_results.md 为准修正
        - 引用未验证 → 标记 [VERIFY] 或替换为已验证文献
        - 统计错误 → 回调 /stat-analysis 重新计算

    对 WARN 问题，以协调者视角决策:
        - 影响较大 → 修复
        - 影响较小 → 记录但保留

    保存状态到 REVIEW_STATE.json
    当前轮次 += 1

if 当前轮次 > MAX_ROUNDS:
    输出最终报告，标注未解决的问题
```

### Step 5: 状态持久化

每轮结束后写入 `REVIEW_STATE.json`：

```json
{
  "current_round": 2,
  "max_rounds": 3,
  "scores": [
    {"round": 1, "score": 58, "grade": "C", "blocks": 3, "warns": 7, "infos": 12},
    {"round": 2, "score": 74, "grade": "B", "blocks": 0, "warns": 4, "infos": 9}
  ],
  "unresolved_blocks": [],
  "unresolved_warns": ["..."],
  "status": "PASSED",
  "timestamp": "2026-03-18T14:30:00"
}
```

## 输出
- `quality_report.md` — 完整质量报告（含各维度得分、问题清单、修复记录）
- `REVIEW_STATE.json` — 审查状态（支持断点恢复）
- 修复后的 `manuscript.md`（如有修改）
- 更新 `progress.md` 中质量审查阶段状态

## 关键规则
1. **BLOCK 不可覆盖**：学术诚信问题没有商量余地，即使用户要求也不跳过
2. **三轮上限**：超过 3 轮仍不通过，输出报告交用户决策，不无限循环
3. **修复后重审**：每次修复后从头重审全部八层，不只审修改的部分
4. **状态可恢复**：REVIEW_STATE.json 确保上下文窗口耗尽后可从中断轮次继续
5. **评分客观**：按矩阵打分，不因"已经改了三轮"就放水
6. **大文件处理**：如果 Write 工具因文件过大失败，立即用 Bash (cat << 'EOF' > file) 分块写入。不要询问用户——直接执行
