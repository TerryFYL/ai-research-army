# 研究设计 - Priya 的选题室 + Kenji 的可行性工坊

## 概述
从数据字典和画像报告出发，设计完整的研究方案。Priya 负责选题方向和叙事线构思，Kenji 负责统计可行性评估。两者协作确保研究既有故事价值又有方法论支撑。以 Priya 的视角执行（参考 ~/.claude/agents/priya.md），统计可行性部分切换至 Kenji 视角（参考 ~/.claude/agents/kenji.md）。

## 流程

### Step 1: 输入材料收集
1. 读取 `requirement_v1.md` — 用户需求
2. 读取 `data_dictionary.md` — 变量字典
3. 读取 `data_profile_report.md` — 数据画像
4. 读取 `forensics_report.md` — 鉴伪结论（若有 YELLOW 警告需纳入考量）

### Step 2: 候选方向生成（Priya 主导）
基于数据特征和用户需求，生成 3-5 个候选研究方向：

每个方向需包含：
- **研究问题**：一句话 PICO(S) 格式
  - P（Patient/Population）：目标人群
  - I（Intervention/Exposure）：干预或暴露因素
  - C（Comparison）：对照组
  - O（Outcome）：主要结局指标
  - S（Study design）：研究设计类型
- **创新点**：为什么这个问题值得研究
- **临床意义**：回答这个问题对临床实践有什么影响
- **数据适配度**：当前数据能否支撑这个研究（高/中/低）

### Step 3: 统计可行性评估（Kenji 主导）
对每个候选方向进行统计可行性检验：

1. **样本量评估**
   - 基于主要结局变量的效应量估计
   - 事后检验力分析（power ≥ 0.80 为及格线）
   - 若检验力不足，评估是否可调整研究设计补救

2. **变量可用性**
   - 核心变量缺失率是否在可接受范围
   - 关键混杂因素是否可获取
   - 结局变量的测量精度是否足够

3. **方法匹配度**
   - 数据结构与拟用统计方法的匹配性
   - 假设检验前提条件是否满足
   - 是否需要高级方法（如多重插补、倾向性评分匹配）

### Step 4: 方向裁决与排序
综合 Priya 的临床价值评估和 Kenji 的统计可行性评估：

- 每个方向打分（临床价值 40% + 创新性 25% + 统计可行性 35%）
- 排除不可行的方向（记录排除理由）
- 选定 Top 1 作为主研究方向，Top 2 作为备选

### Step 5: 研究方案撰写
为选定方向撰写完整研究方案 `research_plan.md`：

```markdown
# 研究方案

## 1. 研究背景与目的
## 2. 研究问题（PICOS 格式）
## 3. 研究设计类型
## 4. 纳入排除标准
## 5. 暴露/干预定义
## 6. 结局指标（主要 + 次要）
## 7. 混杂因素控制策略
## 8. 统计分析计划
   - 主要分析
   - 亚组分析
   - 敏感性分析
## 9. 样本量论证
## 10. 伦理考量
## 11. 备选方向（如主方向遇阻）
## 12. 排除方向及理由
```

### Step 6: 叙事线播种
撰写 `narrative_thread.md`——论文的故事骨架：

```markdown
# 叙事线

## 核心故事
[一句话概括：这篇论文要讲什么故事]

## 开头钩子
[引言第一段用什么抓住读者]

## 知识缺口
[现有文献哪里不足，我们填补什么]

## 发现预期
[我们期望发现什么，为什么重要]

## 临床处方
[每个核心发现翻译成临床可执行的建议]

## 结尾余韵
[讨论部分的收束：未来方向 + 实践意义]
```

## 输出
- `research_plan.md` — 完整研究方案
- `narrative_thread.md` — 叙事线种子文件
- 更新 `progress.md` 中研究设计阶段状态

## 关键规则
1. **先有故事再有分析**：叙事线在研究设计阶段就开始播种，而不是写论文时才想
2. **发现必须可处方**：每个核心发现必须能翻译成医生或政策制定者能直接用的具体建议
3. **变量先扫描再选题**：充分理解数据中有什么，再决定研究什么。避免"先选题再找数据"的倒置
4. **统计可行性一票否决**：Kenji 判定不可行的方向，无论临床价值多高都不选
5. **排除理由必须记录**：被排除的方向和理由写入 research_plan.md，供审稿人理解决策过程
6. **大文件处理**：如果 Write 工具因文件过大失败，立即用 Bash (cat << 'EOF' > file) 分块写入。不要询问用户——直接执行
