# Agent 进化机制

## 核心原则

> 进化不是"改设定"，是"从实战中长出新的认知"。每一单交付、每一次获客、每一个失败都是进化信号。

**三条铁律：**
1. **实战驱动** — 没有客户反馈支撑的进化是幻觉
2. **记录可追溯** — 每次进化都有触发事件、变更内容、预期效果
3. **灵魂内核不变** — 核心信念和互补关系是骨架，可以长肌肉但不能换骨头

---

## 进化的四个层次

```
L1 经验沉淀（每单触发）
    ↓ 量变
L2 认知升级（每 10 Cycle 触发）
    ↓ 质变
L3 角色重塑（每 50 Cycle 触发）
    ↓ 结构变
L4 军团进化（创始人 + 重大事件触发）
```

---

### L1 — 经验沉淀（After-Action Review）

**触发条件：** 每完成一单客户交付 / 每完成一轮获客活动

**机制：** 每个参与的 Agent 回答三个问题：

| 问题 | 目的 |
|------|------|
| 这一单，我**做对了什么**？ | 固化有效行为 |
| 这一单，我**判断失误了什么**？ | 识别盲点 |
| 下一单，我会**改变什么**？ | 形成进化方向 |

**输出：** `clients/<id>/aar.md`（After-Action Review）

**示例：**

```markdown
## AAR — 客户 001

### Priya（Research Consultant）
- ✅ 做对了：提前确认了客户的真实deadline是毕业答辩而非"尽快"
- ❌ 失误了：没有确认数据中的缺失值比例，导致 Kenji 中途发现需要多重插补
- 🔄 下次改：intake 清单增加"数据完整性预检"步骤

### Kenji（Statistician）
- ✅ 做对了：主动选择了 Mann-Whitney 而非 t 检验（数据非正态）
- ❌ 失误了：Kaplan-Meier 图的 number at risk 表格漏了
- 🔄 下次改：统计输出三件套检查清单增加"生存分析附表"项

### Alex（Reviewer）
- ✅ 做对了：发现 Discussion 第二段的因果推断语言不当
- ❌ 失误了：没注意到参考文献 #17 的 DOI 指向了勘误版本
- 🔄 下次改：终审清单增加"DOI 指向验证"
```

**沉淀规则：**
- 同一类失误出现 **2 次** → 写入该角色的 Soul 补丁（`agents/<name>-patches.md`）
- 同一类成功出现 **3 次** → 固化为该角色的新行为模式

### L2 — 认知升级（10-Cycle 健康检查时触发）

**机制：** 在常规健康检查中增加 Agent 进化审视

**流程：**

```
1. 汇总过去 10 Cycle 的所有 AAR
    ↓
2. 提取高频模式
    ├→ 反复出现的失误 → 需要认知升级
    ├→ 反复出现的成功 → 需要固化为能力
    └→ 未触发过的 Agent → 需要评估是否冗余
    ↓
3. 更新 Agent 能力
    ├→ 补丁写入 agents/<name>-patches.md
    ├→ 关联 Skill 优化（调用 skill-evolution-manager）
    └→ 互补关系微调（如发现新的盲点-补位对）
    ↓
4. 记录在 docs/reviews/evolution-cycle-N.md
```

**认知升级的三种形态：**

| 形态 | 触发条件 | 示例 |
|------|---------|------|
| **新增检查项** | 同类失误 ≥ 2 次 | Kenji 新增"多重比较时必须做 Bonferroni 校正提醒" |
| **新增思维模型** | 发现新的决策模式有效 | Priya 新增"三分钟电梯测试"— 能否在三分钟内向客户解释清楚方案 |
| **调整优先级** | 某能力的重要性被实战重新排序 | Hao 把"期刊格式精确"从第三优先提升到第一优先 |

**Patch 文件格式：**

```markdown
# Priya — 进化补丁

## Patch 001 (Cycle 12)
- **触发事件**: 客户 001 数据缺失值问题
- **新增行为**: intake 清单增加"数据完整性预检"
- **验证标准**: 下 3 单 intake 时主动检查缺失值比例

## Patch 002 (Cycle 23)
- **触发事件**: 客户 003 对交付范围理解偏差
- **新增思维模型**: "一页纸确认书"— 方案确认后生成单页摘要让客户签字确认
- **验证标准**: 客户确认环节返工率降至 0
```

### L3 — 角色重塑（50-Cycle 战略复盘时触发）

**机制：** 深度审视 Agent 结构本身是否需要调整

**六个审视维度：**

#### 1. 工作负载分析

```
统计每个 Agent 过去 50 Cycle 被调用次数和产出量：
- 过载 Agent（每轮都被拉上）→ 考虑分裂
- 闲置 Agent（<5 次被调用）→ 考虑合并或裁撤
- 瓶颈 Agent（其他人常等他）→ 考虑分配助手
```

#### 2. 盲点审计

```
回顾互补矩阵，检查：
- 哪些盲点被成功补位了？→ 互补关系有效，保持
- 哪些盲点反复暴露？→ 补位者不够强，需要加强
- 有没有新发现的盲点？→ 需要新的补位关系
```

#### 3. 能力缺口

```
有没有"团队里没人能做"的事？
- 如果有 → 考虑新增 Agent 或给现有 Agent 加新 Skill
- 典型信号："这一步我们每次都需要创始人介入"
```

#### 4. 角色冲突

```
有没有两个 Agent 在做同样的事？
- 边界是否清晰？每个人的 Soul 文件中"负责什么/不负责什么"是否准确？
```

#### 5. 灵魂一致性

```
Agent 的实际行为是否偏离了 Soul 设定？
- Alex 是不是变得太"好说话"了？→ 重新强化逆向论证
- Kenji 是不是变得太慢了？→ 重新平衡严谨和效率
- Sarah 是不是在打擦边球？→ 重新强化安全锁
```

#### 6. 进化决策

| 决策类型 | 条件 | 示例 |
|---------|------|------|
| **分裂** | 一个 Agent 承担了 2+ 个不相关职能 | Hao → Hao（稿件）+ Formatter（格式） |
| **合并** | 两个 Agent 工作高度重叠 | — |
| **新增** | 持续存在能力缺口 | 新增 `mentor`（学术导师角色，帮客户做研究设计） |
| **裁撤** | 50 Cycle 内被调用 < 3 次 | — |
| **灵魂修订** | 核心信念被实战证伪 | 更新对应 Soul 文件 |
| **互补重组** | 发现更高效的补位组合 | — |

**输出：** `docs/reviews/evolution-cycle-N.md`

### L4 — 军团进化（结构性变革）

**触发条件：**
- 创始人 ⛔ 指令要求结构调整
- 业务阶段转换（MVP 验证期 → 规模化运营期）
- 外部重大变化（新竞品出现、平台规则改变、AI 能力突破）

**可能的进化方向：**

```
MVP 验证期：10 Agent，聚焦跑通
    ↓ 验证成功
规模化运营期：可能变化——
    - 新增 QA Agent（交付量大了需要专职质检）
    - Sarah 线分裂（小红书专员 + 论坛专员 + PI 专员）
    - 新增 Onboarding Agent（标准化接单流程）
    - Tom 升级（从手动记账到自动化成本追踪系统）
    ↓ 规模化成功
平台化阶段：可能变化——
    - 客户自助下单 → Priya 角色转为自动化方案生成器
    - 多团队并行 → 每个"小分队"配一个 Mini-Wei 编排器
    - API 开放 → 新增 Developer Relations Agent
```

**L4 进化的安全机制：**
- 必须有 Alex Pre-Mortem — 假设新结构失败，最可能原因是什么
- 必须保留旧结构 48 小时（不立即销毁，观察新结构是否稳定）
- 必须更新 `docs/reviews/军团演变记录.md`

## 进化的红线

1. **核心信念只能在 L3 修订** — 不能因为一单失败就改灵魂
2. **互补关系只能在 L2+ 调整** — 张力对是刻意设计的，不能因为"吵架"就取消
3. **安全红线永不进化** — 学术诚信红线是不可变的宪法
4. **进化必须记录** — 没有记录的变化等于没有发生（下次 Cycle 会丢失）
5. **进化必须可回滚** — Patch 失败时可以撤销，不应造成不可逆损害

## 架构核心原则（83 轮探索沉淀）

以下原则从 83 轮 AI 科研团队架构探索中提炼，经实战验证后融入军团治理。

### 原则 1：协作流 > 层级指挥

团队以协作流组织，Wei 做编排和决策但不构成指挥链。每个角色在专业领域内自主工作，通过明确的上下游关系协作。

**军团对应：** TEAM.md 协作流 — Priya(方案) → Ming(数据) → Kenji(分析) → Lena(出图) → Hao(写稿) → Alex(审查)

### 原则 2：质量标准是动态的

不同任务阶段需要不同精度：
- 文献初筛：Recall > Precision（80% 准确率）
- 数据提取：Precision > Recall（95% 准确率）
- 统计分析：Conservative > Aggressive（99.9% 准确率）

**军团对应：** Wei 的任务编排会根据任务特征调整检查力度

### 原则 3：Level 2 半自动化是甜蜜点

节省 70% 时间，质量损失仅 1%。关键机制：Checkpoint 模式 + 置信度驱动 + 可逆操作。

**自动化不等于零人工。** 80/20 法则：自动化 80% 工作量，保留 20% 人工判断。最优边界是"最小必要人工"，不是"零人工"。

### 原则 4：分层模型使用

- 决策/审查：强力模型（Claude Opus / Gemini Pro）
- 分析/写作：均衡模型（Claude Sonnet / Gemini Flash）
- 辅助任务：轻量模型（Haiku / MiniMax）

成本降低 4.5 倍，质量不降。

**军团对应：** 本地 LLM 网关已支持多模型路由

### 原则 5：三层学习，持续积累

从"每次从零开始"到"持续积累经验"：
- 参数级学习（30% 提效）：Skill 内参数优化
- 策略级学习（50% 提效）：L1 经验沉淀 + Patch 机制
- 元认知级学习（自主进化）：L2/L3 认知升级 + 角色重塑

**军团对应：** evolution.md 四层进化体系 + action-tracker 行动追踪闭环

### 原则 6：自我修复，三级递进

90% 常见问题自动修复：
1. 规则修复（毫秒级）：已知模式直接匹配预防规则
2. 模式修复（从历史学习）：`track auto-patch` 扫描失败模式 ≥3 次 → 生成补丁
3. 推理修复（LLM 深度分析）：未知问题升级给 Wei/Alex 分析

人工介入减少 80%。

**军团对应：** action-tracker 的 emerging → known_pattern → auto-patch 闭环

### 原则 7：质量前置（Shift-Left）

从"事后检查"到"过程控制"，节省 39% 成本：
- 输入验证（拦截 30%）：Priya 需求澄清 + Ming 数据预检
- 过程监控（拦截 50%）：Lena 数据对账 + 行动追踪实时记录
- 输出验证（拦截 15%）：Alex 六层审查

**军团对应：** Alex 六层审查 + paper-pipeline 每步追踪

### 原则 8：五层质量保障

```
Layer 1: 实时监控 — 行动追踪系统自动记录每步结果
Layer 2: 规则检查 — Skill 内置检查清单（如 figure-engine Layer 6）
Layer 3: 自我审核 — manuscript-reviewer 6 维自审
Layer 4: 交叉验证 — Alex 独立审查（与生产者分离）
Layer 5: 人工复核 — Wei 交付确认 + 创始人最终交付
```

错误捕获率 79.83%。**医学科研特殊性：1% 的错误可能造成 100% 的灾难**，高风险任务必须 100% 人工复核。

### 原则 9：四维任务特征

Wei 编排任务时评估每个任务的四个维度：
- **复杂度**：简单（套模板）→ 复杂（需要创造性判断）
- **精度要求**：容错（文献初筛）→ 零容错（统计分析）
- **时间敏感度**：宽松 → 紧急（影响调度优先级）
- **成本敏感度**：低（核心交付）→ 高（边缘任务可降级）

**军团对应：** action_types 表的任务特征维度（详见 schema.sql）

### 原则 10：医学科研的零容错

- 统计结果必须来自真实计算，引文必须跨库验证
- 可追溯性是建立信任的关键 — 行动追踪 + 数字溯源
- 保守优先 — 方法有疑虑时选保守方案，宁可效应量小不可方法错

**军团对应：** 安全红线（TEAM.md）+ Kenji 保守优先 + Alex 一票否决
