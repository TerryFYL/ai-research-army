## Product Value

For medical students, clinicians, and medical researchers, the value here is not “generate some text faster.” It is a full workflow standardization layer:

- less blind trial-and-error
- less rework across data, stats, figures, references, and writing
- less prompt fragmentation
- more reusable research operations for an individual or a lab

---

## What Problem Does It Solve

The highest-value part of research is asking good questions, interpreting findings, and making academic judgments.

But a lot of time is actually spent on:

- cleaning data
- running statistics
- making figures and fixing formatting
- checking citations
- turning outputs into a draft

AI Research Army automates that mechanical chain without taking over the researcher’s judgment.

You provide data and a research direction. It can produce:

| Deliverable | Description |
|--------|------|
| `data_profile_report.md` | data profile |
| `research_plan.md` | research design and statistical plan |
| `analysis_results.md` | structured statistical outputs |
| `figures/` | publication-grade figures |
| `verified_ref_pool.md` | verified reference pool |
| `manuscript.md` | IMRAD draft |
| `quality_report.md` | 8-layer quality report |
| `submission_package/` | journal-ready submission package |
| `delivery/` | final delivery directory |

> The research question, interpretation, clinical meaning, and submission decision remain your responsibility.

## What Is Open-Sourced

This is no longer the earlier lightweight demo.

This repository is now the **public core** of AI Research Army, including:

- 11 agent role definitions
- full module ordering and hard constraints
- `/start-army` orchestration entry
- standalone skills
- templates, registries, and basic validation scripts
- entry docs for Claude Code, Codex-style agents, and Gemini CLI
- multi-CLI adaptation guidance

It does **not** include:

- client data, real project outputs, databases, or logs
- private API keys, gateways, or deployment internals
- commercial auto-loop operations, acquisition SOPs, or payment workflows

In short: **the engine core is open-sourced, not the private business backend**.

## Quick Start

```bash
git clone https://github.com/TerryFYL/ai-research-army.git
cd ai-research-army
bash install.sh
```

Open Claude Code:

```text
/start-army "Investigate the association between sedentary behavior and cardiovascular risk in NHANES 2017-2020"
```

Or call modules directly:

```text
/data-profiler
/data-forensics
/research-design
/stat-analysis
/journal match
/quality-review
/delivery
```

## More Than Prompt Templates

|  | Typical skill collections | AI Research Army |
|---|---|---|
| **Methodology** | Minimal | Full research workflow methodology |
| **Quality gates** | Self-scored outputs | 8-layer review + citation verification + anti-p-hacking checks |
| **Deliverables** | Markdown reports | stats + figures + IMRAD draft + submission package + delivery directory |
| **Traceability** | Weak | numbers and figures are traceable |
| **Reporting compliance** | Rare | STROBE / CONSORT style checks |
| **Resume capability** | Weak | stage-based recovery |
| **Execution engine** | Single CLI | multi-CLI compatible |

## Public-Core Pipeline

`intake → data-explore → data-forensics → research-design → statistics → figures → literature → manuscript → review → submission → delivery`

Key gates:

- `data-profiler` before any real analysis
- `data-forensics` required for raw-data projects
- `statistics` before figures, literature, and manuscript
- `quality-review` is blocking
- `submission_package/` is not the final state; the final state is `delivery/`

See:

- `modules/MODULE_INDEX.md`
- `modules/constraints.yaml`

## Multi-CLI Adaptation

The system is not locked to a single CLI. Each agent can run on the engine that best fits its role.

| Engine | Command | Default Fit | Strength |
|------|------|------|------|
| Claude Code | `claude` | Wei, Kenji, Ming, Hao | deep reasoning, code execution, MCP ecosystem |
| Codex CLI | `codex` | Alex | auditing, review, sandboxed execution |
| Gemini CLI | `gemini` | Lena, Jing | multimodal review, long context |
| OpenCode / Cursor / Cline / Amp / Aider | varies | flexible | team-specific adaptation |

> Claude Code alone is enough for normal use. Multi-engine mode is an advanced option for teams that want cognitive diversity and cross-model review.

## Best-Fit Users

1. Medical students: thesis work, capstones, early research training
2. Clinicians: real clinical questions with limited time for end-to-end execution
3. Medical researchers: labs and teams that want a reusable research operations core

Not a good fit for:

- users with no data who just want a ghostwritten paper
- users unwilling to own research judgment, ethics, and conclusions
- users who want a closed SaaS product instead of a modifiable open core

---

## Not a Ghostwriting Black Box

This project standardizes and automates the mechanical parts of research workflows. It does not replace the researcher’s responsibility for:

- choosing the right question
- checking ethics and permissions
- judging whether conclusions are defensible
- deciding whether and where to submit
